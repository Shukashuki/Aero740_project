## SPIDER — p36-tea 參數實驗（Phase 6 核心）

### Baseline config 解析
從 `config.yaml` 發現 baseline 用的參數與 code 預設不同：
- `temperature=0.1`（預設 0.3）
- `max_num_iterations=32`（預設 16）
- `improvement_threshold=0.001`（預設 0.01）

### 三組實驗結果比較

| 指標 | Baseline | Exp1 (預設 T=0.3,th=0.01) | Exp2 (T=0.3,th=0.001) | Exp3 (T=0.1,th=0.001) |
|---|---|---|---|---|
| opt_steps mean | 9.8 | 1.8 | 5.1 | 4.0 |
| reward mean | **−0.264** | −0.499 | −0.523 | −0.405 |
| reward min | −0.902 | −1.202 | −1.215 | −0.767 |
| tick disc mean | 0.00371 | 0.00360 | 0.00354 | 0.00374 |

### 關鍵發現
1. **Temperature 是最重要的超參數**：T=0.1→−0.405 vs T=0.3→−0.52，改善幅度 > 收緊 threshold
2. **Early stopping threshold 不是瓶頸**：收緊 th 增加 opt_steps 但不改善 reward（假設被否定）
3. **Baseline 未完全復現**（−0.264 vs −0.405），可能是 GPU 非確定性或版本差異
4. **Temperature 的直覺**：T 低=精英獨裁（跟 best sample 走），T 高=民主投票（取 weighted avg）。Contact-rich 任務中精英策略更好，因為 non-convex feasible set 上 convex combination 不封閉

## SPIDER — IK Pipeline 理解（Phase 5 補充）

### 讀完 `spider/preprocess/ik.py`，IK 流程理解：
1. **人手資料（MANO）→ keypoints**：人手骨架被簡化為 8 個點的 (pos, quat)——6 fingertips + palm + object
2. **Mocap constraint IK**：在 MuJoCo 建 mocap bodies 當 target，加 equality constraints（CONNECT for tips, WELD for palm/object），讓 constraint solver 隱式解 IK
3. **不是解析 IK**：沒有 Jacobian pseudo-inverse，直接用物理模擬 settle
4. **多次隨機初始化**：`max_num_initial_guess=8` 避免局部最優
5. **不同部位不同約束強度**：object 最強（solimp=0.001），palm 中等（torque_scale=10），finger tip 最弱

### Roy 的關鍵 insight
- 人手的力學資訊（肌腱、co-contraction、接觸面壓力分佈）在 MANO→keypoints 時完全丟掉
- MuJoCo 用 robot 自己的力學模型「re-couple」這些關係
- **整個 SPIDER 是 cascade of approximations**：每一層丟掉一些東西，下一層用物理模型補回來
  - 人手力學 → 丟掉 → constraint solver 用 robot 力學補
  - IK contact → 丟掉 → MPPI 用 sampling 補
  - MPPI 最優性 → 丟掉 → gain scheduling 用 homotopy 補
- 「一個好的起點 + 基於物理模型的優化算法 = 好的控制序列」
